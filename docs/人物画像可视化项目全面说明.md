# 人物画像可视化项目全面说明

## 1. 文档目的

这份文档用于系统解释当前项目目录中的页面、脚本、数据文件、样式文件和部署文件是如何协同工作的，帮助你后续进行：

- 自己回看项目逻辑
- 向老师、同学或合作方讲解
- 后续维护、扩展和重构
- 判断项目距离“正式产品”还有哪些差距

本文档基于当前仓库实际文件整理，不是抽象模板。

---

## 2. 项目一句话概括

这是一个围绕“相辉学堂学生画像”构建的可视化原型系统，采用：

- 静态 HTML 页面作为前端展示层
- FastAPI 作为后端接口与页面分发层
- CSV 作为学生画像主数据源
- JS 文件作为高校对标数据源
- Render 作为线上部署平台
- GitHub 作为代码托管与部署联动入口

它不是单纯的网页合集，而是一个已经具备“前端页面 + 数据接口 + AI 问答 + 在线访问”的完整演示系统。

### 4.2 用户访问时发生了什么

当用户访问线上链接时，实际上会发生下面的流程：

```text
用户打开网址
  -> Render 上运行的 FastAPI 服务接收请求
  -> backend_formal.py 判断请求的是页面还是接口
  -> 如果是页面，返回对应 html
  -> 如果是接口，读取 CSV 数据并处理
  -> 前端页面用 axios 请求接口
  -> 前端用 ECharts / DOM 渲染可视化结果
```

换句话说，这个项目虽然用了静态 HTML，但并不是“纯静态站”，因为页面内容依赖后端接口。

## 5. 页面体系与内容逻辑

整个网站可分为五个层次：

1. 群体总览层
2. 个人下钻层
3. 高校对标层
4. 资源承接层
5. 方法与制度解释层

### 5.1 群体总览页：`index_formal.html`

这是全项目的主驾驶舱页面，也是最像“系统首页”的页面。

#### 它的作用

- 展示 200 多名学生的总体画像
- 呈现群体类型分布、预警分布、维度均值
- 提供筛选、分组比较和学生目录
- 提供“相辉小助手”群体分析问答
- 把用户继续导向个人页、资源页、政策页和对标页

#### 它主要调的接口

- `/api/formal/overview`
- `/api/formal/warnings`
- `/api/formal/students`
- `/api/formal/group_compare`
- `/api/formal/ai_chat`

#### 页面内部核心逻辑

这个页面的 JS 逻辑主要分为几块：

- `loadOverview()`：加载群体总览 KPI、维度图、类型图、预警图
- `loadWarnings()`：加载重点预警学生列表
- `loadStudents()`：加载当前筛选条件下的学生入口卡片
- `loadAllStudents()`：加载完整学生目录
- `loadGroupCompare()`：按招生类型、专业方向、地区类型等做分组对比
- `sendAssistantMessage()`：向 AI 问答接口发送问题
- `switchGrade()`、`clearFilters()`：切换年级和重置筛选
- `renderDirectory()`：把全部学生目录重新渲染成列表

#### 页面功能结构图

```text
index_formal.html
  ├─ 顶部品牌区与导航
  ├─ KPI 指标卡
  ├─ 维度均值图
  ├─ 学生类型分布图
  ├─ 预警分布图
  ├─ 关键发现摘要
  ├─ 重点学生入口
  ├─ 群体分组比较表
  ├─ 全部学生目录
  └─ AI 助手面板
```

#### 它在整个系统中的位置

它是“从宏观进入系统”的主入口，相当于先让管理者看整体，再决定点进某个学生或某类资源。

### 5.2 个人画像页：`student_formal.html`

这是项目中最有“产品感”的页面之一，负责展示某个学生的完整个体画像。

#### 它的作用

- 根据 URL 参数 `?id=学生学号` 加载单个学生
- 展示个人基本档案
- 展示八大维度雷达图
- 展示成长轨迹图
- 展示风险标签和问题摘要
- 展示建议路径与资源入口
- 提供基于该学生上下文的 AI 解读

#### 它主要调的接口

- `/api/formal/student/{student_id}`
- `/api/formal/ai_chat`

#### 页面内部核心逻辑

- `init()`：页面初始化，拉取学生数据并填充全部模块
- `switchTab()`：切换“优势与关注 / 维度明细 / 行动建议 / 校内资源 / 关键信号”
- `openDimensionDetail()`：打开某个维度详情弹窗
- `openGrowthDetail()`：打开成长轨迹明细弹窗
- `openRiskDetail()`：查看单个风险标签解释
- `openAdviceDetail()`：查看建议详情
- `sendAssistantMessage()`：调用 AI 接口，针对当前学生答疑
- `getStrengthItems()`、`getConcernItems()`：从画像数据里提炼主要优势与关注点

#### 页面内容结构图

```text
student_formal.html
  ├─ 品牌头部与导航
  ├─ 六张 KPI 卡
  ├─ 个人档案卡
  ├─ 八大维度雷达图
  ├─ 成长轨迹图
  ├─ 优势与关注
  ├─ 风险标签
  ├─ 行动建议
  ├─ 校内资源推荐
  ├─ 原始信号解释
  └─ AI 助手面板
```

#### 它在整个系统中的位置

这个页面承担的是“群体识别后，如何落到单个学生”的任务，也就是把群体分析变成个体支持。

### 5.4 多校比较页：`benchmark_compare.html`

这是当前打开的页面，也是对标模块里逻辑最清楚的一页之一。

#### 它的作用

- 根据 URL 参数 `?ids=...` 选择多个高校项目
- 生成比较表
- 生成科研训练条形图
- 生成六维雷达图
- 支持预设样本切换

#### 它的数据来源

- `assets/js/benchmark_data.js`

#### 页面核心逻辑

- `params.get('ids')`：读取 URL 中的项目 id 列表
- `getRows(selectedIds)`：根据 id 列表从 `BENCHMARK_DATA` 取对象
- `render(selectedIds)`：渲染比较表、条形图和雷达图
- `applyPreset()`：把预设组合写回 URL，然后刷新页面

#### 它比较的六个维度

- 导师制
- 科研训练
- 国际交流
- 资助支持
- 选拔机制
- 学科覆盖

#### 页面逻辑图

```text
URL 参数 ids
  -> getRows()
  -> 从 benchmark_data.js 取出多个项目
  -> render()
      -> 生成比较表
      -> 生成科研训练柱状图
      -> 生成六维雷达图
```

#### 它的特点

这一页完全不依赖后端 API，属于“纯前端读取本地 JS 数据”的模式，所以加载会比学生画像页更直接。

### 5.8 方法说明页：`data_methodology.html`

这是“数据可信度说明页面”。

#### 它的作用

- 告诉用户数据从哪里来
- 告诉用户哪些能比、哪些不能比
- 告诉用户页面分值的边界
- 避免用户把 demo 分值误认为真实官方量化评估

#### 这个页面的重要性

企业级系统和普通 demo 的一个重要差别就是：是否主动解释数据边界。这个页面正是在做这件事。

## 6. 后端核心：`backend_formal.py`

这个文件是全项目最关键的核心文件，既是页面分发器，也是 API 服务层。

### 6.1 它负责什么

可以把它理解成五个角色的合体：

1. 页面服务器
2. 静态资源网关
3. 学生画像数据处理器
4. AI 问答代理层
5. 一组分析型 API 的集合

### 6.2 页面分发逻辑

它把下面这些页面声明为公开页面：

- `index_formal.html`
- `student_formal.html`
- `benchmark_overview.html`
- `benchmark_compare.html`
- `benchmark_detail.html`
- `resources_fudan.html`
- `policy_overview.html`
- `data_methodology.html`

因此浏览器访问：

- `/` 会打开首页
- `/benchmark_compare.html` 会打开比较页
- `/student_formal.html?id=20240001` 会打开某个学生页

### 6.3 它处理哪些数据

#### 学生数据

- 从 `formal_student_data.csv` 读取
- 使用 `load_df()` 做缓存
- 如果 CSV 文件更新时间变了，就重新加载

#### 对标数据

- 不由后端处理
- 对标页面直接在前端读 `assets/js/benchmark_data.js`

这意味着当前系统其实存在两条数据路径：

```text
学生画像路径：浏览器 -> FastAPI -> CSV -> JSON -> 页面
高校对标路径：浏览器 -> benchmark_data.js -> 页面
```

### 6.4 后端核心函数

#### 数据与过滤函数

- `load_df()`：读取 CSV 并缓存
- `filter_df()`：按年级、类型、预警、专业方向筛选
- `grade_reference()`：生成同年级参考组

#### 画像加工函数

- `dim_payload()`：把八大维度组织成前端好用的结构
- `risk_payload()`：把风险标签转成带说明的列表
- `build_growth_series()`：生成成长轨迹线
- `build_type_compare()`：生成学生类型对照信息
- `dataset_summary()`：生成群体摘要文本
- `student_context_text()`：生成单个学生的上下文文本，供 AI 使用

### 6.5 主要 API 列表

#### 基础核心接口

- `/api/formal/overview`
  返回群体概览、维度均值、类型分布、预警分布

- `/api/formal/students`
  返回学生列表，可按年级、类型、预警等级、专业方向筛选

- `/api/formal/student/{student_id}`
  返回单个学生完整画像

- `/api/formal/warnings`
  返回重点预警学生列表

- `/api/formal/group_compare`
  返回按分组字段聚合后的比较结果

- `/api/formal/ai_chat`
  返回 AI 助手回答

#### 扩展分析接口

- `/api/formal/mentors`
  以学生类型分组，模拟“导师视角群体画像”

- `/api/formal/mentors/{mentor_id}`
  查看某个导师组的详情

- `/api/formal/students/{student_id}/growth`
  获取学生成长轨迹

- `/api/formal/prediction`
  做风险预测型分析

- `/api/formal/compare`
  做多个学生或多个群体的维度对比

- `/api/formal/churn_risk`
  计算流失风险评分

- `/api/formal/classes`
  按年级和专业方向生成班级视图

### 6.6 后端 API 组织方式

```text
backend_formal.py
  ├─ 页面路由
  ├─ 文档与静态文件路由
  ├─ 数据读取与缓存
  ├─ 数据加工函数
  ├─ 群体画像接口
  ├─ 个人画像接口
  ├─ 风险与比较接口
  ├─ 导师与班级接口
  └─ AI 问答接口
```

### 6.7 AI 问答逻辑

AI 问答是通过 `/api/formal/ai_chat` 实现的。

#### 流程图

```text
前端输入问题
  -> POST /api/formal/ai_chat
  -> 后端读取问题、页面类型、学生ID
  -> 生成群体摘要 / 学生上下文
  -> 拼接 system_prompt + user_prompt
  -> 调用兼容 OpenAI 或 Anthropic 的模型接口
  -> 返回答案
```

#### 当前支持方式

- OpenAI 兼容接口
- DeepSeek 兼容接口
- Anthropic messages 兼容接口

#### 如果没配置 API key 会怎样

后端不会崩，而是返回一段“离线说明版回答”。这是一个很好的容错设计。

## 7. 学生数据生成逻辑：`formal_data_generate.py`

这个文件决定了学生画像数据是怎么“造出来”的。

### 7.1 它的核心作用

生成模拟学生样本，并输出到 `formal_student_data.csv`。

### 7.2 它做了哪些事情

#### 先定义基础变量池

比如：

- 生源省份
- 专业方向
- 招生类型
- 地区类型
- 中学层级
- 性别
- 学生类型

#### 再生成原始行为与心理信号

例如：

- 学术身份认同
- 科学家志向
- 专业理解
- 目标清晰度
- 学习动力
- 心理健康
- 压力水平
- 好奇心
- AI 使用频率
- 导师关系
- 同伴支持
- 科研参与程度
- FDUROP 参与情况
- P/NP 使用情况

#### 然后计算八大维度

它会用一组加权公式，把原始信号组合成：

- 学术志趣
- 学业韧性
- 创新人格
- 学习动力
- 科研参与
- 导学关系
- 环境满意度
- 心理健康

#### 再做类型识别和风险识别

- `classify_student()`：判断学生属于哪类画像类型
- 风险标签包括心理、导师缺失、志趣不足、科研不足、外驱主导、AI 使用不足、过载等
- 还会生成预警等级：`red / yellow / green / none`

#### 最后生成建议文本

- 学术建议
- 心理建议
- 技能建议
- 资源建议
- 路径建议

### 7.3 数据生成流程图

```text
基础属性随机生成
  -> 原始行为/心理信号生成
  -> 八大维度加权计算
  -> 百分位与 z-score 计算
  -> 学生类型分类
  -> 风险标签识别
  -> 建议路径生成
  -> 输出 CSV
```

### 7.4 当前数据规模

当前 `formal_student_data.csv` 中实际读取到的数据为：

- 220 条学生样本
- 131 个字段

这已经足够支撑原型级展示，但还不是生产级数据体系。

## 8. 学生主数据文件：`formal_student_data.csv`

这个文件是人物画像模块的核心数据载体。

### 8.1 字段类型大致可分为六层

## 9. 高校对标数据逻辑：`assets/js/benchmark_data.js`

这个文件是高校对标模块的数据底座。

### 9.1 为什么它放在 JS 里

因为当前对标模块追求：

- 本地直接打开也能看
- 不依赖额外后端接口
- 改动快
- 前端页面可以直接读取

所以这里用了最简单直接的形式：在 JS 文件中定义 `BENCHMARK_DATA` 数组。

### 9.2 每条对标数据包含什么

每个高校项目对象大致包含：

- 学校 id
- 学校名称
- 项目名称
- 学校层级
- 地理区域
- 项目类型
- 目标学生
- 培养目标
- 选拔方式
- 导师制分值与说明
- 科研训练分值与说明
- 国际交流分值与说明
- 资助支持分值与说明
- 选拔机制分值
- 学科覆盖分值
- 标签
- 对外链接

### 9.3 被哪些页面使用

- `benchmark_overview.html`
- `benchmark_compare.html`
- `benchmark_detail.html`

### 9.4 逻辑意义

这个模块和学生画像模块是平行关系，不是上下级关系。

```text
学生画像模块回答：我们自己的学生现在是什么状态？
高校对标模块回答：外部高校项目通常怎么做培养设计？
```

前者偏内部识别，后者偏外部参照。

## 12. 部署逻辑：GitHub + Render

### 12.1 当前部署链路

```text
本地修改代码
  -> git add / commit / push
  -> GitHub 仓库更新
  -> Render 检测到 main 分支变化
  -> Render 自动重新部署
  -> 用户通过 onrender 链接访问最新版本
```

### 12.2 配置文件

- `requirements.txt`：告诉 Render 安装哪些 Python 包
- `render.yaml`：告诉 Render 如何启动服务

### 12.3 启动方式

线上和本地都依赖：

```text
uvicorn backend_formal:app --host 0.0.0.0 --port $PORT
```

### 12.4 为什么线上首次打开慢

因为使用的是 Render 免费实例，空闲后会休眠；再次访问时需要冷启动。

## 13. 文件之间是怎么连起来的

### 13.1 主链路一：学生画像链路

```text
formal_data_generate.py
  -> 生成 formal_student_data.csv
  -> backend_formal.py 读取 CSV
  -> index_formal.html 调 overview/students/warnings/group_compare
  -> student_formal.html 调 student/{id}
```

### 13.2 主链路二：高校对标链路

```text
benchmark_data.js
  -> benchmark_overview.html 读总览
  -> benchmark_detail.html 读单项目
  -> benchmark_compare.html 读多项目比较
```

### 13.3 主链路三：AI 助手链路

```text
用户提问
  -> 前端 axios POST /api/formal/ai_chat
  -> backend_formal.py 拼接数据上下文
  -> 调外部 AI 模型接口
  -> 返回文字回答
```

### 13.4 主链路四：线上部署链路

```text
本地项目目录
  -> GitHub
  -> Render
  -> 对外访问链接
```

## 14. 这个项目当前最核心的设计逻辑

如果用一句更系统的话来总结，这个项目的设计逻辑是：

### 14.1 先识别

通过群体页和个人页回答：

- 学生整体是什么状态
- 某个学生是什么状态

### 14.2 再解释

通过方法页、政策页和 AI 助手回答：

- 为什么这样判断
- 这个分值代表什么
- 风险和优势该如何理解

### 14.3 再承接

通过资源导航页回答：

- 判断之后接什么资源
- 管理者和学生分别能做什么

### 14.4 再外部参照

通过高校对标模块回答：

- 别的高校是怎么做的
- 相辉学堂可以向哪些机制学习

所以这个项目并不是只做了“可视化”，而是尝试把：

`识别 -> 解释 -> 支持 -> 对标`

串成了一条连续逻辑。

## 16. 当前项目的局限

- 学生数据是模拟数据，不是真实生产数据
- 对标数据目前是本地 JS 写死，不是后台维护
- 大量前端逻辑直接写在 HTML 中，后期维护成本会升高
- 还没有用户登录、角色权限、后台录入、审核流
- 没有真正的数据库，CSV 只是临时数据载体
- 阶段文档和 benchmark schema 目前仍有占位性质
- 线上部署依赖免费实例，稳定性和启动速度有限
